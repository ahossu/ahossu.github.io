DWMShield — Kernel-Mode Window Capture Exclusion on Windows

A research proof-of-concept that calls win32kfull!GreProtectSpriteContent directly from a kernel driver to mark any window as excluded from screen capture — making it invisible to every user-mode recording API without touching SetWindowDisplayAffinity.

1. Background — the documented capture exclusion path

Windows 10 introduced SetWindowDisplayAffinity with the flag WDA_EXCLUDEFROMCAPTURE (value 0x11). A process can call it on one of its own windows and that window will disappear from the output of every screen-capture API in user mode — BitBlt, PrintWindow, the Graphics Capture API, OBS, everything. The excluded window appears black or is simply absent in the captured frame, depending on the capturing method.

The catch is that SetWindowDisplayAffinity only works on windows owned by the calling process, and the protection state can be queried with GetWindowDisplayAffinity. Both facts make it unsuitable for anything that needs to apply the exclusion from outside the target window's process, or that needs it to be undetectable from user mode.

The question I started with: is this exclusion purely a DWM-side policy, or is it enforced deeper in the graphics stack?

2. The research question

The public API path is well understood. The interesting part is what happens below it. SetWindowDisplayAffinity is a thin wrapper in user32.dll that ultimately dispatches to the kernel through NtUserSetWindowDisplayAffinity. From there it reaches win32kfull.sys — the kernel-mode component that manages window objects.

The goal was to find the internal routine responsible for actually applying the capture-exclusion flag, call it directly from a driver, and confirm that the result is indistinguishable from the documented path as far as DWM and all user-mode capturers are concerned.

3. Reversing win32kfull.sys to find GreProtectSpriteContent

I loaded win32kfull.sys in IDA and traced the call chain from NtUserSetWindowDisplayAffinity down. The path goes through a handful of internal window-management routines before landing in GreProtectSpriteContent — the function that tells the DWM compositor to apply the capture-exclusion state to the window's sprite.

It is not exported and has no documented prototype. Its address can be found in a live kernel session with WinDbg:

lkd> x win32kfull!GreProtectSpriteContent
fffff806`12345678 win32kfull!GreProtectSpriteContent

The function takes two arguments: an HWND identifying the target window, and a DWORD flags value. Passing 0x11 mirrors exactly what the documented path does when called with WDA_EXCLUDEFROMCAPTURE. After the call, the DWM marks the window's sprite as excluded and stops compositing it into any captured surface.

Because GreProtectSpriteContent is not exported from win32kfull.sys, its address must be found at runtime. KASLR randomises the module base on every boot, so a hardcoded offset would break after a reboot. More on this below.

4. How the driver works

The driver creates a character device \Device\HideWindow with a symbolic link \DosDevices\HideWindow, accessible from user mode as \\.\HideWindow. A small user-mode companion (protect.exe) sends a single IOCTL containing the HWND of any window — it does not need to be owned by the calling process.

The driver's IRP_MJ_DEVICE_CONTROL handler receives the HWND, calls GreProtectSpriteContent at the hardcoded address with the flag 0x11, and returns. That's the entire critical path:

// driver IRP_MJ_DEVICE_CONTROL handler (simplified)
HWND hwnd = *(HWND *)inputBuffer;
typedef NTSTATUS (*GRE_PROTECT)(HWND, DWORD);

GRE_PROTECT fn = (GRE_PROTECT)GRE_PROTECT_SPRITE_CONTENT_ADDR;
fn(hwnd, 0x11);   // 0x11 = WDA_EXCLUDEFROMCAPTURE

After this call the target window is excluded from every user-mode capture surface. OBS, the Windows Graphics Capture API, BitBlt/PrintWindow — all of them render the window as absent or black, identical to the result of calling SetWindowDisplayAffinity through the documented path. The difference is that GetWindowDisplayAffinity called from another user-mode process cannot reliably detect that the protection came from a kernel driver rather than from the window's own process.

Installing and running it requires test-signing mode:

bcdedit /set testsigning on
Restart-Computer

sc.exe create HideWindow type= kernel binPath= C:\Drivers\HideWindow.sys
sc.exe start HideWindow
protect.exe   # sends IOCTL with HWND of target window

5. KASLR and the address problem

The single biggest limitation is KASLR. GreProtectSpriteContent is not exported, so there's no dynamic resolution path without additional work — pattern scanning the loaded module in kernel memory, or maintaining an offset table per Windows build. For a research PoC the address is simply hardcoded as a #define and must be updated after each reboot:

#define GRE_PROTECT_SPRITE_CONTENT_ADDR 0xfffff806`12345678ULL

A more robust implementation would resolve the address at driver load time by scanning for a known byte signature in the win32kfull image. That's a straightforward extension but outside the scope of this PoC — the goal here was to confirm the mechanism, not to build a production-grade tool.

6. Result and takeaway

The experiment confirms that the capture-exclusion mechanism is genuinely enforced inside the kernel, inside DWM's compositor path. There is no user-mode component that could be patched or bypassed to recover the captured content — if GreProtectSpriteContent has been called with the right flag, DWM simply does not include that window in any captured surface.

The corollary is that any security model depending on user-mode APIs to apply or detect this kind of protection has a fundamental ceiling: a kernel driver can apply the same state without going through the documented path, and the user-mode detection APIs will not see a meaningful difference.

The full source is on GitHub: ahossu/DWMShield.

7. Limitations

  • Address hardcoding. The GreProtectSpriteContent address must be updated manually after every reboot. Function signatures can also change across Windows updates.
  • Windows 10 / 11 only. The internal function exists in those versions. It was tested on build 19041 (Windows 10 2004).
  • No HWND validation. The driver passes the HWND directly to the internal function without any sanity check. A malformed handle could cause unpredictable behaviour in the kernel.
  • Test signing required. The driver needs testsigning mode or a valid EV certificate. Neither is suitable for deployment on locked-down systems.
This is a research proof-of-concept. The code is intentionally minimal — it demonstrates the mechanism, not a deployable implementation.
