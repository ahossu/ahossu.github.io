DWMShield — Kernel-Mode Window Capture Exclusion on Windows

A kernel driver PoC that calls win32kfull!GreProtectSpriteContent directly to make any window invisible to every user-mode screen recorder, without going through SetWindowDisplayAffinity.

1. Starting point

SetWindowDisplayAffinity with WDA_EXCLUDEFROMCAPTURE (flag 0x11) hides a window from all user-mode screen capture — BitBlt, PrintWindow, OBS, the Windows Graphics Capture API, all of it. The captured frame either shows the window as solid black or skips it entirely depending on the method used.

Two things about the documented API bothered me. First, it only works on windows owned by the calling process. Second, the protection state is readable via GetWindowDisplayAffinity, so anything checking for it can detect it. I wanted to see what was actually enforcing the exclusion below the API — and whether calling that internal mechanism directly would produce the same result without those constraints.

The full source and build instructions are on GitHub: ahossu/DWMShield.

2. Tracing the call chain in win32kfull.sys

SetWindowDisplayAffinity is a thin wrapper in user32.dll. It goes through NtUserSetWindowDisplayAffinity into the kernel, where win32kfull.sys handles everything related to window objects and the DWM compositor.

I loaded win32kfull.sys in IDA and traced the call chain down from NtUserSetWindowDisplayAffinity. A few layers in, it lands in GreProtectSpriteContent — the routine that tells DWM to mark a window's sprite as excluded from any captured surface. It's not exported and has no documented prototype. Its address in a live kernel session:

lkd> x win32kfull!GreProtectSpriteContent
fffff806`12345678 win32kfull!GreProtectSpriteContent

It takes two arguments: an HWND and a DWORD flags value. Passing 0x11 — the same value as WDA_EXCLUDEFROMCAPTURE — produces exactly the same DWM behavior as the documented path. After the call, the window's sprite is excluded from every captured surface, regardless of which process owns the window.

3. The driver

The driver creates \Device\HideWindow with a symbolic link at \DosDevices\HideWindow (\\.\HideWindow from user mode). A small companion binary, protect.exe, sends a single IOCTL with the target HWND — it doesn't need to own the window.

The IRP_MJ_DEVICE_CONTROL handler reads the HWND out of the input buffer and calls GreProtectSpriteContent:

HWND hwnd = *(HWND *)inputBuffer;

typedef NTSTATUS (*GRE_PROTECT)(HWND, DWORD);
GRE_PROTECT fn = (GRE_PROTECT)GRE_PROTECT_SPRITE_CONTENT_ADDR;
fn(hwnd, 0x11);

That's the entire critical path. After that call, OBS, the Graphics Capture API, BitBlt/PrintWindow — none of them can capture the target window, same as if the window's own process had called SetWindowDisplayAffinity. Installing it requires test-signing mode:

bcdedit /set testsigning on
Restart-Computer

sc.exe create HideWindow type= kernel binPath= C:\Drivers\HideWindow.sys
sc.exe start HideWindow
protect.exe

4. KASLR

GreProtectSpriteContent is not exported, so its address can't be resolved at load time without additional work — pattern scanning the loaded image in kernel memory, or maintaining a per-build offset table. For this PoC the address is hardcoded as a #define and needs to be updated after every reboot:

#define GRE_PROTECT_SPRITE_CONTENT_ADDR 0xfffff80612345678ULL

The current address for any given boot can be read with WinDbg:

lkd> x win32kfull!GreProtectSpriteContent

Pattern scanning at driver load time would fix this, but it's outside the scope of what this PoC was trying to answer.

5. What it confirms

The exclusion is enforced entirely inside the DWM compositor path in the kernel — there's no user-mode component involved in the actual filtering. A user-mode recorder can't work around it, and calling GetWindowDisplayAffinity from another process gives no reliable indication that the protection came from a kernel driver rather than from the window's own process.

It also shows that WDA_EXCLUDEFROMCAPTURE is not just a hint — it's a hard enforcement point inside DWM, applied at the sprite level before any captured surface is composed.

6. Limitations

  • KASLR. The GreProtectSpriteContent address changes every reboot and must be updated manually. Its signature can also change across Windows updates.
  • Windows 10 / 11 only. Tested on build 19041 (Windows 10 2004). The internal function exists in those versions but there's no guarantee about future builds.
  • No HWND validation. The driver passes the HWND directly to the internal function without any sanity checks. A malformed handle could cause unpredictable behaviour in the kernel.
  • Test signing required. The driver is test-signed and requires testsigning mode — not suitable for deployment on a locked-down system.
This is a PoC built to answer a specific question about how DWM enforces capture exclusion. The code is intentionally minimal.
