CourierDrop (OSC) — 4-Stage Android RE

A courier logistics app with four layers of secrets: an obfuscated dispatch code, a JNI callback that hides its output, anti-debug guards protecting encrypted notes, and an HMAC-signed local HTTP verifier that hands out the final flag.

Android • Reverse Engineering OSC regional 2026 Challenge by Alexandru Hossu

  • Stage 2 — native attestation via JNI callback (FLAG 2)

Stage 1 — dispatch code → session token (FLAG 1)

The first screen asks for a "dispatch code". Type the wrong thing and you get Invalid dispatch code. Contact dispatch. — generic, no hints at all. The validation logic is in a heavily obfuscated Kotlin function that applies a series of deterministic transforms: normalization, reversal, XOR with a lookup table, and a checksum comparison against a hardcoded digest.

There's no plain hash you can reverse-lookup. I traced through the transform chain statically and worked backwards to find an input that produces the expected digest. You can also hook the validation function with Frida and brute-force the PIN space — it turns out to be small enough for that to be fast.

Once I had the right dispatch code, the session still wasn't just a flag sitting in plaintext. The code is used as a KDF input to decrypt blob1 — a byte array embedded in the APK. Out comes a session token, which is FLAG 1, and which also gets stored in EncryptedSharedPreferences as the live session credential.

Everything downstream requires a valid session. The app checks for it on every screen and rejects you silently if it's missing or expired — so there's no way to skip this stage.

Stage 2 — native attestation via JNI callback (FLAG 2)

Stage 2 is the package scan flow. The UI shows a progress indicator — "Validating label…", "Checking chain-of-custody signature…", "Device attestation in progress…" — and then marks the stop as Verified. FLAG 2 never appears on screen, which was the first hint something interesting was happening under the surface.

What actually happens: the app calls into libnative.so with the session token from Stage 1. The native side derives a key from that token, decrypts blob2 stored in .rodata, and produces an attestation string. That attestation is FLAG 2.

The catch — it's not returned as a return value. The native code calls back into Java via JNI: either onScanSuccess(attestation) on a callback interface, or directly into a repository class like RouteRepository.saveAttestation(attestation). The UI just shows "Verified" and the flag disappears into memory.

Two good interception points with Frida:

  • Hook the Java method that receives the callback — read the string argument directly
  • Hook NewStringUTF in the JNI bridge — every string the native side creates passes through here
Java.perform(() => {
    // option 1: hook the Java callback
    const RouteRepo = Java.use("com.courierdrop.RouteRepository");
    RouteRepo.saveAttestation.implementation = function(att) {
        console.log("[*] attestation:", att);
        return this.saveAttestation(att);
    };
});

Stage 3 — anti-debug bypass + encrypted locker (FLAG 3)

The "Secure Notes / Locker" section contains sensitive delivery instructions for each stop. Before showing anything real, the app runs Guard.check(). If that returns bad news, you get sanitized placeholder notes — plausible-looking but completely useless.

Guard.check() looks for:

  • Debug.isDebuggerConnected()
  • TracerPid in /proc/self/status
  • Frida-related substrings in /proc/self/maps

I bypassed it by patching the smali. Decompile with apktool, find the branch that gates on the guard result, flip it, rebuild and sign. With the guard out of the way, key3 is derived from the Stage 2 attestation (FLAG 2 feeds directly here), blob3 is decrypted, and the locker receipt — FLAG 3 — shows up in the Diagnostics section.

If you'd rather not patch smali, hooking Guard.check at runtime to always return false works too — just make sure Frida itself doesn't trip the detection before you hook it.

Stage 4 — HMAC proof + local verifier (FLAG 4)

The final screen is "Complete Delivery". Fill in the form fields and the app constructs a POD (proof of delivery) JSON payload, signs it, and sends it to a local HTTP server that the app spins up itself on a random 127.0.0.1 port.

The signing works like this:

nonce = random 16 bytes
canonical = nonce || sha256(payload_json)
K = KDF(locker_receipt)   ← FLAG 3 feeds in here
X-Nonce: base64(nonce)
X-Proof: base64(HMAC-SHA256(K, canonical))

The local verifier (POST /verify) reconstructs the canonical payload, recomputes the HMAC, compares, and — if everything matches — responds with a receipt JSON containing audit_token. That's FLAG 4.

Two ways to get there. The easy path: hook OkHttp's network interceptor, capture the full request including headers, replay it to the local server port (find the port by hooking the server startup or reading it from Diagnostics). The more thorough path: reverse the key derivation from locker_receipt, build the canonical payload yourself, sign it, and send your own POST. I went with the second approach — more work, but more satisfying.

Each flag is derived from the previous one: dispatch code → session token → attestation → locker receipt → audit token. Break one link and everything after it stays locked. The chain is the whole point of the challenge.
